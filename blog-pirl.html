Teaching a 4B Model to Chain Git Operations Through Pure RL: A Debugging Story

Training Qwen3-4B to discover multi-turn git tool chains entirely from reward signal — no demonstrations, no SFT. Six reward function versions, four failure modes, and the fix that finally worked.

Training a language model to use tools is usually done with supervised fine-tuning — you collect demonstrations of correct tool use and train the model to imitate them. We took a different approach: pure reinforcement learning with no demonstrations. The model had to discover how to chain git operations (status → log → show → add → commit) entirely from reward signal.

This is the story of how we got a 4-billion-parameter model (Qwen3-4B-Instruct) from broken to near-perfect on multi-turn git tool use — and the four things that broke along the way.

The Setup

Environment

We built a sandboxed git environment with 13 tools the model can call:

CategoryTools
Readgit_status, git_log, git_show, git_diff, git_diff_staged, git_diff_unstaged, git_branch
Writegit_add, git_commit, git_checkout, git_create_branch, git_reset, git_pull

Each episode drops the model into a fresh, deterministic git repository (seeded from the question text) and asks it to complete a task. There are four sandbox templates:

  • clean: 5 commits, clean working tree
  • dirty: 4 commits + modified tracked file + untracked scratch.py
  • staged: 4 commits + staged but uncommitted changes
  • multi_branch: 3 branches (main, feature-a, feature-b) with independent commits

Episodes

100 episodes across 4 difficulty levels:

LevelTools RequiredExample
L1 (20)1 tool"Check the repository status"
L2 (21)2 tools"Get the HEAD commit hash from the log and show that commit"
L3 (28)3-4 tools"Show staged diff, then find and display the two most recent commits"
L4 (31)4-5+ tools"Switch to feature-a, get its HEAD hash, return to main, create a branch from feature-a's HEAD"

Episodes are split roughly 50/50 between read (report information) and write (mutate the repository). Read episodes use dynamic fragment matching — placeholders like $head_commit_hash are resolved against the actual sandbox state at runtime, so the model can't guess answers.

Training Setup

  • Model: Qwen3-4B-Instruct-2507 (4B parameters)
  • Algorithm: GRPO (Group Relative Policy Optimization) with LoRA
  • Platform: Prime Intellect's TOPLOC distributed RL infrastructure
  • Batch size: 256, rollouts per example: 16
  • Max tokens: 1024, temperature: 0.9

GRPO works by sampling multiple rollouts per episode, computing rewards, then normalizing within each group: advantage = (r - mean(r)) / std(r). The model learns to do more of what scored above average and less of what scored below.

v0.3.0: Variance Starvation

Our first real training run looked promising for about 30 steps, then collapsed.

Step 1:   Reward 0.96
Step 30:  Reward 0.99
Step 50:  Reward 0.0312
Step 51:  Reward 0.0312
...
Step 68:  Reward 0.0312

What happened: The reward function was scoring tool responses, not the model's final answer. Since git commands return deterministic output, even random tool calls produced reward ~0.96. The model quickly hit 0.99+ on all 16 rollouts within each group.

When all rollouts score nearly the same, std(r) ≈ 0. GRPO divides by std, so the advantages become enormous noisy values → the gradient signal degrades → the model's policy drifts → it stops making tool calls entirely → reward collapses to 0.0312 (the format-gate minimum for a single partial match).

The 19-step dead zone (steps 50-68) was stuck at exactly 0.0312 — the model had completely lost the ability to make tool calls and was getting minimal partial credit from text that happened to contain a matching fragment.

Root cause: Variance starvation. When the reward function doesn't discriminate between rollouts, GRPO has nothing to learn from.

v0.4.0: The Plateau

We redesigned the reward function with three components:

  1. Format gate: No tool calls → reward 0.0 (hard cutoff)
  2. Tool bonus: 0.1 × fraction of successful tool calls
  3. Outcome score (0.0–1.0):
    • Read episodes: fraction of expected fragments found in the final assistant message only
    • Write episodes: fraction of expected state conditions satisfied (branches exist, clean working tree, correct commit message, etc.)

Key fix: only scoring the final message meant the model had to actually understand and report what it found, not just parrot tool output.

We also added dynamic fragments — $head_commit_hash, $commit_count, etc. — resolved against the actual sandbox at episode setup time. This prevented the model from memorizing static answers.

We enabled online difficulty filtering to focus training on episodes the model hadn't mastered yet.

Result: The reward flatlined at 0.68 for 265 steps. Then crashed.

Step 1:   Reward 0.68
Step 100: Reward 0.68
Step 200: Reward 0.68
Step 265: Buffer crash — "No environments left with examples"

Diagnosing the plateau: We decoded the reward value. With the tool bonus formula, a reward of 0.68 means:

  • Read episodes: ~0.1 (tool bonus) + ~0.58 (outcome) ≈ 0.68 (partial fragment matching)
  • Write episodes: ~0.1 (tool bonus) + 0.0 (outcome) ≈ 0.1

The average was 0.68 because read episodes scored ~0.8+ while write episodes scored exactly 0.1. The model had learned to use read tools effectively but never learned to call write tools (git_add, git_commit, git_create_branch).

Why write episodes stalled: On write episodes, the model would call read tools (git_status, git_log) — earning the 0.1 tool bonus — but never attempted git_add or git_commit. Since all 16 rollouts for a write episode scored the same 0.1, there was no within-group variance → no gradient → no learning. The same variance starvation problem, just at 0.1 instead of 0.99.

v0.5.0–v0.6.0: Buffer Drain

We tried two fixes:

v0.5.0: Added a write-tool bonus — if the model calls any write-specific tool (git_add, git_commit, git_checkout, git_create_branch, git_reset), it gets a 0.2 bonus instead of 0.1 × tool_success_rate. This creates a gradient: rollouts that attempt write operations score higher than those that only read.

We also tried a phased curriculum: Phase 1 trains on read-only episodes first, Phase 2 adds writes.

v0.5.1: The reads-only Phase 1 worked but saturated quickly. Moving to Phase 2 with difficulty filtering enabled caused a crash:

RuntimeError: No environments left with examples

The difficulty filter was draining episodes faster than the model could learn them. An episode with all rollouts scoring 1.0 gets marked "easy" and removed. An episode with all rollouts scoring 0.1 gets marked "hard" and removed. Eventually, no episodes remain.

v0.6.0: Same crash. The filter's easy/hard thresholds couldn't handle the bimodal reward distribution (reads ≈ 1.0, writes ≈ 0.1).

v0.6.1: The Fix

Two changes:

  1. Disabled difficulty filtering entirely. The write-tool bonus provides enough within-group variance for GRPO to learn without needing to filter episodes.
  2. Kept the write-tool bonus at 0.2. This is the key insight: on write episodes, a rollout that calls git_add scores 0.2 while one that only calls git_status scores 0.1. That 0.1 difference gives GRPO signal to reinforce the write-tool behavior, which then leads to actual successful writes (outcome score > 0) in later rollouts.

Result:

Step 1:   Reward 0.68
Step 50:  Reward 0.85
Step 100: Reward 0.95
Step 150: Reward 0.99
Step 200: Reward 1.00
Step 250: Reward 0.99
Step 300: Reward 0.99

The model went from 0.68 to near-perfect in ~150 steps, then sustained 0.97–1.00 for the remaining 150 steps. Total training time: approximately 3 hours.

What the Model Learned

By step 150, the model had discovered these multi-step tool chains with zero demonstrations:

Read chains: git_log → extract commit hash → git_show <hash> → report findings. The model learned to parse structured git output, extract specific values (commit hashes, file names), and thread them into subsequent tool calls.

Write chains: git_status → git_add . → git_commit -m "message" → git_create_branch <name>. The model learned the correct ordering of git operations and that you must stage before committing.

Cross-branch navigation: git_checkout feature-a → git_log → extract info → git_checkout main → git_create_branch <name>. The model learned to navigate between branches, gather information, and return — a 4-5 tool chain requiring state tracking across turns.

Why This Is Interesting

Model size: 4B parameters. Most published tool-use training uses SFT with curated demonstrations on 70B+ models. We used pure RL on a model 17× smaller.

No demonstrations: The model had no examples of correct tool use. It discovered git_log → extract hash → git_show entirely from the reward signal telling it "you reported the right commit hash" vs "you didn't."

Multi-turn reasoning: This isn't single-turn function calling. The model makes 3-5 sequential tool calls, each depending on the output of the previous one. It learned to parse structured text output, extract specific values, and use them as arguments in subsequent calls.

The reward design was everything: We spent 6 versions getting the reward right. The final insight — that a small bonus (0.2 vs 0.1) for attempting the right category of tool is enough to bootstrap learning — is general. Any RL setup where one action type is harder than another can benefit from a "you're on the right track" bonus that creates within-group variance for policy gradient methods.

Lessons Learned

  1. Variance starvation is the silent killer of GRPO training. If all rollouts in a group score the same, there's no gradient — no matter how high or low the score. Watch for flat reward curves and decode the actual per-episode-type rewards.
  2. Difficulty filtering can backfire with bimodal reward distributions. If half your episodes are "easy" (reward ≈ 1.0) and half are "hard" (reward ≈ 0.1), the filter removes everything. The model never gets a chance to learn the hard tasks because they're filtered before it can start making progress.
  3. Small reward bonuses create large learning effects. The 0.1 difference between "called a write tool" (0.2) and "only called read tools" (0.1) was enough to bootstrap an entire capability. You don't need large reward differences — you need any difference that correlates with the right behavior direction.
  4. Decode your reward values. When we saw 0.68, we didn't just see "needs improvement." We decomposed it: 0.68 = average of read episodes (~0.8+) and write episodes (~0.1). This told us exactly what was broken and why.
  5. Start simple, add complexity only when you understand the failure mode. We tried difficulty filtering, phased curricula, reads-only training — all before understanding that the core issue was within-group variance on write episodes. The fix was simpler than any of the attempted solutions.

What's Next

  • Evaluation on held-out L5 tasks: Merge conflicts, cherry-pick, stash/pop — git operations the model has never seen. Does it generalize from the tool-use patterns it learned, or did it memorize 100 task templates?
  • Scale the model: Qwen3-8B or 14B with the same curriculum. If 4B hits 1.0, a larger model may generalize better to unseen scenarios.
  • Harder curricula: Interactive rebase, conflict resolution, multi-remote workflows — tasks that require genuine reasoning, not just tool chaining.

Trained on Prime Intellect using TOPLOC distributed RL. Environment built with verifiers.
